# 명령: 리팩토링 (refactor)

아래 빈칸을 채워서 Claude에게 전달하세요.

---

## 리팩토링 대상
- **파일/영역**: <예: features/board/ 전체, posts 라우트>
- **목표**: <가독성 / 중복 제거 / 레이어 분리 / 타입 강화 / 성능 중 무엇>
- **알려진 제약**: <건드리면 안 되는 것, 공개 API 시그니처 등>

## 절대 원칙
> **동작은 절대 바뀌면 안 된다 (behavior-preserving).** 입력/출력/사이드이펙트가 동일해야 한다. 기능 변경이 필요하면 멈추고 사용자에게 알린다.

## Claude가 따라야 할 절차
1. **현황 파악**: 대상 코드와 그 호출부를 읽고, 무엇이 어떻게 쓰이는지 요약.
2. **계획 제시**: 단계별 변경 계획을 먼저 보여주고, 큰 변경이면 사용자 승인 후 진행.
3. **작게 쪼개기**: 한 번에 하나의 관심사만. 예) "중복 fetch 로직을 services로 추출" → 그다음 "타입 정리".
4. **패턴 정렬**: 이 프로젝트의 표준 구조로 맞춘다.
   - 컴포넌트의 직접 네트워크 호출 → `services/`로 추출.
   - 중복 타입 → `@maxt/shared`로 통합.
   - 거대한 컴포넌트/핸들러 → feature 폴더 / 라우트 단위로 분리.
   - 반복 UI → `components/ui`의 `Cy*` 프리미티브로 치환.
5. **검증**: 변경 전후 동작이 같음을 어떻게 확인했는지 설명. [harness/checklist.md](../../harness/checklist.md) 점검.

## 하지 말 것
- 요청 범위를 벗어난 "겸사겸사" 변경.
- 스타일 규칙 변경(세미콜론/따옴표 등).
- 공개 함수/라우트 시그니처를 말없이 바꾸기.
